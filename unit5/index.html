Definition / Theory

Both Servlets and JSP (JavaServer Pages) are server-side Java technologies used to build dynamic web applications, but they approach the problem from opposite directions.

  • Servlet: A pure Java class that extends HttpServlet. It contains Java code with HTML embedded inside out.println() statements. It is best for processing logic and control.
  • JSP: An HTML page with Java code embedded inside special tags (<% %>). It is best for the presentation/view layer.

Key differences:

  • Coding style: Servlet = Java with HTML inside; JSP = HTML with Java inside.
  • Translation: A JSP is automatically translated into a Servlet by the container, then compiled. So JSP runs as a Servlet internally.
  • Ease of writing: JSP is easier for designers since the page looks like HTML; Servlets are harder for large UIs.
  • Maintenance: JSP separates presentation, so it is easier to maintain; Servlets mix logic and markup.
  • Best role (MVC): Servlet acts as the Controller, JSP acts as the View.

In practice both are used together: the Servlet handles the request and business logic, then forwards data to a JSP that renders the response. This combination is the foundation of the JSP-based MVC pattern.

Definition / Theory

In the MVC (Model–View–Controller) pattern, a Servlet plays the role of the Controller — the central component that receives every request, decides what to do, and chooses the response page.

Functions of a Servlet as Controller:

  • Receives requests: All client requests first arrive at the Servlet through doGet() / doPost().
  • Reads input: It extracts form data using request.getParameter().
  • Calls the Model: It invokes business logic / JavaBeans / DAO classes to process data or query the database.
  • Stores results: It puts data into request/session scope using request.setAttribute().
  • Selects the View: It forwards control to a JSP using RequestDispatcher.forward(), which renders the HTML.
  • Controls navigation: Based on logic it can choose different JSP pages (success / error).

The key idea is separation of concerns: the Servlet (Controller) never generates HTML itself; it only coordinates. The Model holds data and logic, the View (JSP) displays it. This makes the application cleaner, reusable and easier to maintain, because a designer can change the JSP without touching Java logic, and a developer can change logic without touching the HTML.

Code Example
LoginController.java 
@WebServlet("/login")                       // maps URL to this controller
public class LoginController extends HttpServlet {
    protected void doPost(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {

        // 1) READ input from the request
        String user = req.getParameter("username");
        String pass = req.getParameter("password");

        // 2) CALL the Model (business logic)
        UserService service = new UserService();
        boolean ok = service.validate(user, pass);

        // 3) STORE result + 4) SELECT the View
        if (ok) {
            req.setAttribute("name", user);          // pass data to View
            RequestDispatcher rd = req.getRequestDispatcher("welcome.jsp");
            rd.forward(req, res);                     // forward to JSP (View)
        } else {
            req.setAttribute("error", "Invalid login");
            req.getRequestDispatcher("login.jsp").forward(req, res);
        }
    }
}

Definition / Theory

The Spring Framework is a lightweight, open-source Java application framework used to build enterprise applications. It is built around two core principles: IoC (Inversion of Control) and DI (Dependency Injection), which together make code loosely coupled, testable and maintainable.

Spring Architecture (layered modules):

  • Core Container: The heart of Spring. Contains Core, Beans, Context and SpEL modules. The BeanFactory / ApplicationContext (the IoC container) creates and manages objects called beans.
  • Data Access / Integration: JDBC, ORM, JMS, Transactions — simplifies database access and transaction management.
  • Web Layer: Spring Web, Spring MVC, WebSocket — for building web applications and REST APIs.
  • AOP (Aspect Oriented Programming): Adds cross-cutting concerns (logging, security, transactions) without changing business code.
  • Test: Supports unit and integration testing with JUnit/TestNG.

Key features:

  • IoC & Dependency Injection: The container injects dependencies instead of objects creating them — reduces tight coupling.
  • AOP: Modularises cross-cutting concerns.
  • Lightweight & modular: Use only the modules you need.
  • Transaction management: Consistent, declarative transactions.
  • MVC framework: Powerful web layer.
  • Integration: Works with Hibernate, JPA, JMS, REST.

In short, Spring provides a complete infrastructure so developers focus on business logic while the framework manages object lifecycle, wiring and transactions.

Definition / Theory

Spring Boot is a project built on top of the Spring Framework that makes it fast and easy to create stand-alone, production-ready Spring applications with minimal configuration. Its motto is "convention over configuration."

Key features:

  • Auto-Configuration: Automatically configures beans based on the libraries on the classpath — no manual XML.
  • Starter Dependencies: Single dependencies like spring-boot-starter-web pull in everything needed for that feature.
  • Embedded Server: Includes a built-in Tomcat / Jetty — run the app with java -jar, no external server.
  • Spring Boot Actuator: Production endpoints for health, metrics, monitoring.
  • Spring Initializr: Web tool to generate a ready project skeleton.
  • No XML / boilerplate: Uses annotations and sensible defaults.

Advantages:

  • Rapid development: Less configuration, faster startup of projects.
  • Standalone deployment: A single executable JAR with embedded server.
  • Microservice ready: Ideal for building independent REST microservices.
  • Reduced boilerplate: Auto-configuration removes repetitive setup code.
  • Easy testing & monitoring via Actuator and test starters.

In short, Spring Boot lets developers focus on business logic rather than configuration, which is why it is the modern standard for Java web and microservice development.

Definition / Theory

JSP scripting elements let you embed Java inside an HTML page. This question focuses on two of them: directives and expression tags.

1. Directives <%@ ... %> give instructions to the JSP container at translation time. There are three:

  • page directive <%@ page %> — sets page properties like language, contentType, import, errorPage.
  • include directive <%@ include file="..." %> — statically includes another file (header/footer) at translation time.
  • taglib directive <%@ taglib %> — declares a custom/JSTL tag library.

2. Expression tag <%= expression %> evaluates a Java expression, converts the result to a String, and writes it directly into the output. There is no semicolon inside an expression tag.

Together they cover the two most common tasks: directives configure the page (imports, content type) and expression tags display dynamic values (dates, calculations, variables) inside the HTML. Because directives run at translation and expressions run at request time, a single JSP can both set up its environment and output live data.

JSP Scripting Elements

↓
Declaration <%! ... %> fields / methods
↓
Scriptlet <% ... %> Java statements
↓
Expression <%= ... %> prints value to output

Definition / Theory

MVC (Model–View–Controller) is a design pattern that separates an application into three layers so that presentation, logic and data are independent. In a JSP-based web application this is called Model 2 architecture.

  • Model: Represents the data and business logic — usually JavaBeans, POJOs and DAO classes that talk to the database.
  • View: The presentation layer — implemented with JSP pages that display data to the user. The View contains no business logic.
  • Controller: A Servlet that receives all requests, calls the Model, and selects the correct View.

Working flow (Model 2):

  • 1. The browser sends a request to the Controller (Servlet).
  • 2. The Controller reads parameters and calls the Model for processing.
  • 3. The Model executes business logic / database operations and returns results.
  • 4. The Controller stores the result (setAttribute) and forwards to a JSP.
  • 5. The JSP (View) renders the HTML response back to the browser.

Why MVC? It gives separation of concerns, reusability, parallel development (designers edit JSP, developers edit logic) and easy maintenance. The older Model 1 put logic inside the JSP itself, which mixed concerns; Model 2 (MVC) fixes that by adding a Servlet controller.

            throws ServletException, IOException {

        req.setAttribute("student", s);         // store data for the View
        req.getRequestDispatcher("result.jsp")  // select the View
           .forward(req, res);                  // forward to JSP
    }
}

Definition / Theory

A Spring Boot web application is created in a few standard steps and follows a fixed Maven directory structure. This question tests whether you can show the layout, the pom.xml, the main class and a controller, then run it.

Steps to develop:

  • 1. Generate the project from Spring Initializr with the spring-boot-starter-web dependency.
  • 2. pom.xml declares the parent and the web starter (pulls in Tomcat + Spring MVC).
  • 3. The main class annotated @SpringBootApplication bootstraps the app and starts embedded Tomcat.
  • 4. A @RestController (or @Controller) defines request mappings.
  • 5. Run with mvn spring-boot:run or java -jar target/app.jar; open http://localhost:8080.

Standard directory structure: source code lives under src/main/java, configuration (like application.properties) and templates under src/main/resources, and tests under src/test/java. The build output JAR is placed in target/. Spring Boot's auto-configuration means you do not write any server or dispatcher setup — the framework wires everything from the classpath, so the running result is a self-contained web server responding on port 8080.

Definition / Theory

These three are related but distinct. Spring MVC and Spring Boot are both built on top of the core Spring Framework.

1. Spring Framework — the foundation. A lightweight container providing IoC / Dependency Injection, AOP, transactions and integration. Features: loose coupling, modular design, AOP, declarative transactions, wide integration (Hibernate, JPA, JMS).

2. Spring MVC — the web module of Spring. It implements the Model–View–Controller pattern for web apps. The central piece is the DispatcherServlet (front controller) that routes requests to @Controller methods and resolves views (JSP/Thymeleaf). Features: annotation-based controllers (@RequestMapping), flexible view resolution, form binding & validation, REST support.

3. Spring Boot — a tool / convention layer that makes Spring (and Spring MVC) easy to set up. Features: auto-configuration, starter dependencies, embedded Tomcat, Actuator, no XML — produces a runnable JAR.

Relationship: Spring is the base engine; Spring MVC adds the web layer; Spring Boot removes the configuration pain and packages everything for production. You typically build a Spring MVC web app inside Spring Boot, sitting on the Spring core.

Definition / Theory

This question combines three JSP scripting elements in one page: directives, declarations and scriptlets.

  • Directive <%@ ... %> — configures the page at translation time (e.g. <%@ page import="..." %>).
  • Declaration <%! ... %> — declares instance variables and methods that become members of the generated Servlet class. Used to define reusable methods/fields.
  • Scriptlet <% ... %> — contains Java statements that go inside the _jspService() method and execute on every request.

Key difference (declaration vs scriptlet):

  • A declaration defines methods/fields outside the service method (class level) — written once.
  • A scriptlet runs code inside the service method — executes each request.

So a declaration is ideal for a helper method or counter variable, while a scriptlet is used for logic that runs per request (loops, conditions, calling methods). Combined with an expression tag <%= %> to print results, these elements let a single JSP both define logic and produce dynamic output — though best practice limits scriptlets in favour of JSTL/EL.

Where Each Element Lands in the Servlet
<%@ page %> Directive → imports / class config
↓
<%! int count; String fn(){...} %> Declaration → class-level members (once)
↓
<% for(...) {...} %> Scriptlet → inside _jspService() (per request)
↓
<%= value %> Expression → writes to response
Code Example
counter.jsp 
<%-- DIRECTIVE: page configuration + import --%>
<%@ page language="java" contentType="text/html; charset=UTF-8"
         import="java.util.*" %>

<%-- DECLARATION: class-level field + method (defined once) --%>
<%!
    int visits = 0;                       // instance variable
    int square(int n) { return n * n; }   // reusable method
%>

<html>
<body>
  <h2>JSP Directives, Declarations & Scriptlets</h2>

  <%-- SCRIPTLET: runs on every request --%>
  <%
      visits++;                            // increment counter each request
      out.println("<p>Page visits: " + visits + "</p>");

      // a loop printing squares using the declared method
      for (int i = 1; i <= 5; i++) {
          out.println("<p>Square of " + i + " = " + square(i) + "</p>");
      }
  %>
</body>
</html>

Definition / Theory

A basic Spring MVC application follows the Model–View–Controller pattern using Spring's web module. The central component is the DispatcherServlet (front controller), which receives every request and delegates it.

Working flow:

  • 1. The request reaches the DispatcherServlet.
  • 2. It consults the HandlerMapping to find the matching @Controller method.
  • 3. The Controller runs logic, adds data to the Model, and returns a view name.
  • 4. The ViewResolver maps that name to an actual page (e.g. welcome.jsp).
  • 5. The View renders the Model data and the response goes back to the browser.

Key annotations: @Controller marks the class as a web controller; @RequestMapping / @GetMapping map URLs to methods; Model carries data to the view; @RequestParam reads query parameters. The Controller returns a logical view name, not HTML, keeping presentation separate. This separation is exactly what MVC provides — the Controller handles logic, the View (JSP/Thymeleaf) handles display, and Spring wires them through the DispatcherServlet and ViewResolver.

Code Example
HelloController.java + view config 
// ===== CONTROLLER =====
package com.example.demo;
import org.springframework.stereotype.Controller;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.*;

@Controller
public class HelloController {

    @GetMapping("/welcome")                  // maps GET /welcome
    public String welcome(@RequestParam(defaultValue = "Guest") String name,
                          Model model) {
        model.addAttribute("name", name);    // add data to the Model
        model.addAttribute("course", "Web Technology");
        return "welcome";                     // logical VIEW name
    }
}
welcome.jsp (View) + application.properties 
<%-- ===== VIEW: welcome.jsp ===== -->
<html>
<body>
  <h2>Welcome to Spring MVC!</h2>
  <!-- EL reads attributes added by the Controller's Model -->
  <p>Hello, ${name}!</p>
  <p>Course: ${course}</p>
</body>
</html>

<!-- ===== application.properties : ViewResolver config ===== -->
<!-- maps "welcome" -> /WEB-INF/views/welcome.jsp -->
spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp

Exam Tips
  • Always name the 5 components: DispatcherServlet, HandlerMapping, Controller, ViewResolver, View.
  • The Controller returns a logical view name (a String), not HTML — the ViewResolver maps it to a JSP.
  • Mistake: returning HTML from @Controller. Use @RestController only when returning raw data, not views.
