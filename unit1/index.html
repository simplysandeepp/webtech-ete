Definition / Theory

Client–Server Architecture is a distributed computing model in which tasks are divided between two parties: the client (which requests services) and the server (which provides services). It is the backbone of almost every modern web application.

  • Client: Any device or program (web browser, mobile app) that initiates a request. It handles the presentation layer — the user interface.
  • Server: A powerful machine that listens for requests, performs processing/business logic, accesses databases, and returns a response.
  • Request–Response cycle: The client sends an HTTP request; the server processes it and returns an HTTP response (HTML, JSON, files).
  • Stateless communication: HTTP is stateless — each request is independent. State is maintained using cookies, sessions, or tokens.

Architectures are classified by tiers: a 2-tier model connects client directly to a database server, while a 3-tier model adds a middle application server (presentation → logic → data). The 3-tier model improves scalability, security and maintainability because each layer can be developed and scaled independently. Key advantages include centralized data control, easy maintenance, and resource sharing; the main drawbacks are server overload under heavy traffic and a single point of failure if the server crashes.

client-server-demo.html 
<!DOCTYPE html>
<html>
<head><title>Client-Server Demo</title></head>
<body>

  <button onclick="getData()">Request Data from Server</button>
  <p id="result">Click the button to send a request...</p>

  <script>
    // CLIENT sends a request to the SERVER (a public test API)
    function getData() {
      const out = document.getElementById('result');
      out.textContent = '⏳ Sending HTTP request to server...';

      // fetch() performs the request-response cycle over HTTP
      fetch('https://jsonplaceholder.typicode.com/users/1')
        .then(response => response.json())   // SERVER returns JSON response
        .then(data => {
          out.textContent = '✅ Server replied: ' + data.name +
                            ' (' + data.email + ')';
        })
        .catch(err => out.textContent = '❌ Error: ' + err);
    }
  </script>
</body>
</html>

Definition / Theory

A protocol is a set of rules that governs how data is formatted, transmitted and received between computers on a network. The Internet runs on a protocol stack (TCP/IP model) where each layer has dedicated protocols.

  • HTTP (HyperText Transfer Protocol): Transfers web pages between browser and server. Works on port 80. Stateless request-response. Example: GET /index.html.
  • HTTPS: Secure HTTP using SSL/TLS encryption on port 443. Protects data like passwords.
  • FTP (File Transfer Protocol): Uploads/downloads files between client and server on ports 20/21.
  • SMTP (Simple Mail Transfer Protocol): Sends emails (port 25). Receiving uses POP3 (110) or IMAP (143).
  • TCP/IP: TCP guarantees reliable, ordered delivery; IP handles addressing & routing of packets.
  • DNS (Domain Name System): Translates domain names (google.com) into IP addresses (port 53).

When you type a URL: DNS resolves the name → TCP opens a connection → HTTP/HTTPS requests the page → IP routes the packets back. Together these protocols make the web reliable and interoperable.

Definition / Theory

Before HTML5, multimedia required plugins like Flash. HTML5 introduced native <audio> and <video> elements so browsers can play media directly — no plugins needed.

  • <audio> embeds sound files. Supported formats: MP3, OGG, WAV.
  • <video> embeds video. Supported formats: MP4, WebM, OGG.
  • <source> lets you provide multiple formats; the browser picks the first it supports — improving compatibility.

Important attributes:

  • controls — shows play/pause/volume UI.
  • autoplay — starts automatically (often needs muted).
  • loop — repeats playback.
  • muted — starts without sound.
  • poster (video only) — placeholder image before playback.
  • preload — hints how much to buffer (none/metadata/auto).

These tags make media accessible, responsive and SEO-friendly, and they work consistently across modern browsers and mobile devices.

Definition / Theory

HTML5 provides two ways to draw graphics in the browser: Canvas and SVG.

Canvas (<canvas>) is a bitmap/raster drawing surface controlled by JavaScript. You get a 2D context and issue drawing commands (lines, arcs, images). It is resolution-dependent — once drawn, shapes become pixels and "forget" they exist (immediate mode). Ideal for games, charts, image editing.

SVG (Scalable Vector Graphics) is vector-based and written in XML. Each shape is a DOM element you can style with CSS and attach events to (retained mode). Being vectors, SVG is resolution-independent — it scales to any size without blur. Ideal for logos, icons, diagrams.

Key differences:

  • Canvas: pixel-based, JS-driven, fast for many objects, not scalable, no DOM per shape.
  • SVG: shape-based, XML/DOM, fully scalable, CSS-stylable, slower with thousands of objects.

Rule of thumb: choose Canvas for dynamic pixel-heavy rendering (games) and SVG for crisp, interactive, scalable graphics (UI icons, infographics).

Definition / Theory

XML (eXtensible Markup Language) is a markup language designed to store and transport data in a self-descriptive, platform-independent way. Unlike HTML (which displays data), XML carries data with custom tags you define yourself.

XML structure rules (well-formedness):

  • Must have exactly one root element.
  • Every tag must be properly closed and correctly nested.
  • Tags are case-sensitive.
  • Attribute values must be quoted.
  • An optional prolog: <?xml version="1.0" encoding="UTF-8"?>.

XML Namespaces prevent naming conflicts when XML from different vocabularies is combined (e.g. two <table> tags meaning different things). A namespace is declared with the xmlns attribute and a unique URI, usually with a prefix:

  • xmlns:prefix="URI" binds a prefix to a namespace.
  • Elements then use prefix:element to stay unambiguous.
  • A default namespace (xmlns="URI") applies to all unprefixed child elements.

XML is the foundation of SOAP, RSS, SVG, XHTML and configuration files, making it essential for data exchange between systems.

XML Tree Structure
  • <bookstore> (root)
    • <book>
      • <title>
      • <author>
      • <price>

Hierarchical (tree) model: one root → child elements → leaf text nodes

Code Example
catalog.xml 
<?xml version="1.0" encoding="UTF-8"?>
<!-- Namespaces: 'lib' and 'store' avoid <item> conflicts -->

  </lib:item>

    <store:title>USB Cable</store:title>

Definition / Theory

The DOM (Document Object Model) represents an HTML page as a tree of objects. An event is a signal that something happened (a click, key press, page load). Event handling is the process of detecting these events and running JavaScript in response.

Three ways to attach events:

  • Inline HTML attribute: <button onclick="fn()"> — simple but mixes HTML & JS.
  • DOM property: element.onclick = fn; — only one handler per event.
  • addEventListener(): el.addEventListener('click', fn) — best practice, allows multiple handlers.

Common events: click, mouseover, mouseout, keydown, keyup, submit, load, change.

Event flow has three phases: Capturing (top→target), Target, and Bubbling (target→top). The event object (event) carries details like event.target, and methods preventDefault() (stop default action) and stopPropagation() (stop bubbling). Event delegation uses bubbling to handle many child elements with a single listener on the parent — efficient for dynamic lists.

Event Flow Diagram
1. Capturing Phase document → target (top-down)
↓
2. Target Phase event reaches the clicked element
↓
3. Bubbling Phase target → document (bottom-up)
Code Example
events.html 
// 1. addEventListener (recommended)
const btn = document.getElementById('myBtn');
btn.addEventListener('click', function (event) {
  console.log('Clicked element:', event.target.id);
  alert('Button was clicked!');
});

// 2. Mouse events
btn.addEventListener('mouseover', () => btn.style.background = 'gold');
btn.addEventListener('mouseout',  () => btn.style.background = '');

// 3. Form submit + preventDefault
document.getElementById('f').addEventListener('submit', function (e) {
  e.preventDefault();           // stop page reload
  console.log('Form intercepted, not reloading');
});

Definition / Theory

A portfolio website showcases a person's skills, projects and contact details. Using HTML5 semantic elements gives the page meaningful structure that improves accessibility and SEO.

Key semantic elements used:

  • <header> — top section with name/logo & navigation.
  • <nav> — navigation menu links.
  • <section> — thematic grouping (About, Projects).
  • <article> — self-contained content like a project card.
  • <aside> — side content (skills/links).
  • <figure> / <figcaption> — media with captions.
  • <footer> — copyright & contact.

Multimedia (image/video) makes the page engaging, while a contact <form> with inputs (text, email, textarea) lets visitors reach out. Semantic tags replace generic <div>s so screen readers and search engines understand the page layout — this is the main reason examiners ask for semantic markup.

Page Layout (Semantic Structure)
<header> + <nav>
<section id="about"> (photo + bio)
<section id="projects"> → <article> cards
<section id="contact"> → <form>
<footer>

Definition / Theory

A travel blog shares stories, photos, videos and locations of trips. It combines several HTML5 features into one page:

  • Navigation menu (<nav>): links to Home, Destinations, Gallery, Contact.
  • Semantic structure: each trip becomes an <article> inside a <section>, with <header> and <footer>.
  • Multimedia: <img> photos, <video> clips, and <figure>/<figcaption> captions.
  • Embedded maps: an <iframe> from Google Maps shows the trip location directly on the page.

The <iframe> element embeds another HTML document (like a map or YouTube video) inside the current page using its src attribute. Google Maps provides an embed link that you paste into the iframe src. This makes content interactive without leaving the blog. Using semantic tags plus multimedia and maps creates a rich, accessible, mobile-friendly blog — exactly what the question asks you to demonstrate.

Blog Page Structure
<header> — Blog title + <nav> menu
<article> — Trip story + <figure> photo/video
<iframe> — Embedded Google Map
<footer> — Social links

Definition / Theory

A responsive web page adapts its layout to any screen size (mobile, tablet, desktop). HTML5 + the viewport meta tag + flexible CSS achieve this. This question tests whether you can combine all core HTML elements correctly.

  • Viewport meta: <meta name="viewport" content="width=device-width, initial-scale=1"> — essential for responsiveness.
  • Semantic tags: header, nav, main, section, footer.
  • Lists: <ul>/<ol>/<li> for menus and points.
  • Tables: <table>, <tr>, <th>, <td> for tabular data.
  • Hyperlinks: <a href> for navigation.
  • Images: <img> with max-width:100% to scale.
  • Forms: <form> with inputs and a submit button.

Responsiveness is reinforced with relative units (%, rem), flexbox/grid, and media queries. The result is a single page that reads well on a phone and a monitor — the definition of a modern, accessible HTML5 document.

responsive.html

  <meta charset="UTF-8">

        <tr><th>Item</th><th>Price</th></tr>
        <tr><td>Laptop</td><td>₹55,000</td></tr>
      </table>

        <button>Go</button>
      </form>

</body>

Definition / Theory

A multi-page company website consists of several linked HTML pages — typically Home, About, Services, Contact — sharing a common navigation menu. Each page is a separate .html file and links connect them with relative href values.

Core building blocks:

  • Consistent <header> + <nav> on every page so users can move between pages.
  • Semantic content tags: <main>, <section>, <article>, <aside>.
  • Forms: a <form> on the Contact page (name, email, message).
  • Embedded multimedia: a banner <video> or product <img> on the Home page.
  • Footer: address, copyright, social links repeated across pages.

Linking pages: <a href="about.html">About</a>. Keeping the header/nav/footer identical creates a professional, cohesive feel. This question checks navigation between multiple physical pages (not just sections) plus the same semantic + form + media skills — so demonstrate at least two pages and the links that join them.
