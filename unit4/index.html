Definition / Theory

Apache Tomcat is an open-source Servlet container and web server that implements the Jakarta Servlet, JSP, and WebSocket specifications. It provides a pure Java HTTP web server environment for running Java web applications.

Core architectural components:

  • Catalina (Servlet Container): The heart of Tomcat — manages the servlet life cycle, handles request dispatching, and enforces security constraints. It contains the Engine, Host, Context and Wrapper containers in a hierarchical pipeline.
  • Coyote (HTTP Connector): The HTTP connector that listens on a TCP port (default 8080) for incoming HTTP requests, parses them into HttpServletRequest objects, and sends HttpServletResponse objects back to the client.
  • Jasper (JSP Engine): The JSP compiler that translates .jsp files into Java Servlet source code, then compiles them into .class files for execution. Jasper handles the entire JSP-to-Servlet translation pipeline.
  • Cluster: Supports session replication and load balancing across multiple Tomcat instances for high availability and scalability.

The architecture follows a layered, modular design: the Connector receives raw HTTP requests, hands them to the Engine, which routes them through the Host → Context → Wrapper hierarchy to the target Servlet. The response flows back along the same path. This separation allows Tomcat to serve static content efficiently while also supporting dynamic Servlet/JSP processing, security realms, and clustering — all in a single server process.

Definition / Theory

Client-side scripting refers to code that runs on the user's browser (client machine), while server-side scripting runs on the web server. Both generate dynamic web content but operate in fundamentally different environments with distinct trade-offs.

Client-side scripting (e.g. JavaScript):

  • Execution location: In the browser after the page is downloaded. The source code is visible to the user.
  • Speed: Very fast after loading — no network round trip for each action. UI updates are instant.
  • Capabilities: Can manipulate the DOM, handle events, validate forms, create animations, and make AJAX calls. Cannot access server files or databases directly.
  • Security: Less secure — code is exposed and can be viewed/modified by the user.
  • Example languages: JavaScript, TypeScript, jQuery, React.

Server-side scripting (e.g. Java Servlets, JSP, PHP):

  • Execution location: On the web server before the response is sent to the browser. The source code is never visible to the client.
  • Speed: Slower per interaction — each action requires a request-response cycle over the network.
  • Capabilities: Can access databases, files, session data, and perform business logic. Cannot directly manipulate the browser DOM.
  • Security: More secure — code stays on the server; only output HTML is sent to the client.
  • Example languages: Java Servlets, JSP, PHP, Python (Django), Node.js, ASP.NET.

Modern web applications combine both: server-side code handles data processing and storage (the "backend"), while client-side JavaScript provides a responsive, interactive user interface (the "frontend"). This separation is the foundation of the client-server and MVC architectures.

        req.getRequestDispatcher("done.jsp").forward(req, res);
    }

Definition / Theory

The Servlet life cycle defines the sequence of methods invoked by the servlet container from the moment a servlet is loaded till it is destroyed. It is managed entirely by the container (e.g. Apache Tomcat's Catalina).

Three main phases:

  • 1. Initialization (init()): Called once when the servlet is first loaded into memory. The container creates the servlet instance and calls init(ServletConfig). This is where you open database connections, read configuration parameters, or set up resources. Throwing ServletException here prevents the servlet from being placed into service.
  • 2. Service (service() → doGet()/doPost()): Called for each client request. The container creates ServletRequest and ServletResponse objects and passes them to service(), which examines the HTTP method and delegates to doGet(), doPost(), etc. This method executes the servlet's core logic — reading parameters, processing data, and writing the response.
  • 3. Destruction (destroy()): Called once when the container decides to remove the servlet (server shutdown, reload). Use this to clean up — close DB connections, release file handles, stop background threads. After destroy() returns, the servlet is eligible for garbage collection.

The container also manages loading (either eagerly via load-on-startup in web.xml or lazily on first request) and multithreading: since the servlet is a singleton, the service() method runs concurrently for multiple requests — the developer is responsible for thread safety when accessing shared instance variables.

Code Example
LifeCycleServlet.java 
import java.io.*;
import javax.servlet.*;
import javax.servlet.http.*;

@WebServlet("/lifecycle")
public class LifeCycleServlet extends HttpServlet {

    // 1. INIT — called once when servlet is first loaded
    @Override
    public void init(ServletConfig config) throws ServletException {
        super.init(config);
        System.out.println("=== INIT: Servlet loaded. " +
            "Opening DB connection...");
        // e.g., DriverManager.getConnection(...);
    }

    // 2. SERVICE — called per-request, dispatches to doGet/doPost
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {
        System.out.println("=== SERVICE: Handling GET request");
        res.setContentType("text/html");
        res.getWriter().println("<h2>Servlet Life Cycle Demo</h2>");
    }

    // 3. DESTROY — called once before removal
    @Override
    public void destroy() {
        System.out.println("=== DESTROY: Closing DB connection...");
        // e.g., connection.close();
    }
}

  • Memorise the three methods: init(), service() (or doGet/doPost), destroy().
  • Emphasise that init() and destroy() run once; service() runs per request. This is the most common exam distinction.
  • Mistake: forgetting that the servlet is a singleton — instance variables are shared across concurrent requests, so thread safety matters.

Definition / Theory

JDBC (Java Database Connectivity) is a Java API that provides a standard way to connect Java applications with relational databases. It follows a driver-based architecture that decouples application code from database-specific implementation.

Architecture layers (top-down):

  • Java Application: Your Java code that uses the JDBC API to execute SQL queries and process results. The application interacts only with the abstract JDBC interfaces — it does not know which database vendor is underneath.
  • JDBC API (java.sql package): A set of standard interfaces — DriverManager, Connection, Statement, PreparedStatement, CallableStatement, ResultSet. These are the same regardless of the database.
  • JDBC Driver Manager: DriverManager class manages a list of registered JDBC drivers. When you call getConnection(), it iterates through registered drivers and picks the one that can handle the given JDBC URL.
  • JDBC Drivers: Vendor-specific implementations of the JDBC interfaces. There are four types: Type 1 (JDBC-ODBC bridge — deprecated), Type 2 (native API, partly Java), Type 3 (network protocol, pure Java), and Type 4 (pure Java, direct-to-database — the most common, e.g. MySQL Connector/J, PostgreSQL JDBC).
  • Database: The actual DBMS (MySQL, Oracle, PostgreSQL, etc.) that stores and manages data.

Key point: JDBC provides database independence. Changing the database vendor only requires swapping the driver JAR and changing the connection URL — your Java code stays the same. This is the fundamental advantage of the JDBC layered architecture.

Code Example
JDBC connection example 
import java.sql.*;

public class JdbcDemo {
    public static void main(String[] args) throws Exception {
        // 1. Load the JDBC driver (optional in modern JDBC 4+)
        Class.forName("com.mysql.cj.jdbc.Driver");

        // 2. Get connection via DriverManager
        String url = "jdbc:mysql://localhost:3306/mydb";
        String user = "root";
        String pass = "password";
        Connection conn = DriverManager.getConnection(url, user, pass);

        // 3. Create a Statement
        Statement stmt = conn.createStatement();

        // 4. Execute query and get ResultSet
        ResultSet rs = stmt.executeQuery("SELECT * FROM students");

        // 5. Process results
        while (rs.next()) {
            System.out.println(rs.getInt("id") + " - " +
                               rs.getString("name"));
        }

        // 6. Close resources (always in reverse order)
        rs.close();  stmt.close();  conn.close();
    }
}

Definition / Theory

In Java Servlets, every HTTP request is encapsulated in an HttpServletRequest object and every response is written via an HttpServletResponse object. These two objects are the fundamental interface between the servlet and the HTTP protocol.

HttpServletRequest — receiving data from the client:

  • getParameter(name) — reads a single query/form parameter value.
  • getParameterValues(name) — reads multiple values for the same parameter (e.g. checkboxes).
  • getMethod() — returns GET, POST, PUT, etc.
  • getRequestURI() / getQueryString() — URL details.
  • getSession() — returns the current HttpSession.
  • getHeader(name) — reads request headers.
  • getReader() / getInputStream() — reads the request body.

HttpServletResponse — sending data to the client:

  • setContentType(type) — sets the MIME type (e.g. text/html, application/json).
  • getWriter() — returns a PrintWriter for writing text output.
  • getOutputStream() — returns a ServletOutputStream for binary output (images, files).
  • sendRedirect(url) — sends a 302 redirect to another URL.
  • setHeader(name, value) — sets response headers.
  • setStatus(sc) — sets the HTTP status code (200, 404, 500, etc.).

The typical flow: the servlet reads parameters from the request, processes them (calling business logic / DAO layers), sets response content type, and writes the HTML (or JSON, XML) to the response writer. In an MVC pattern, the servlet forwards to a JSP instead of writing HTML directly, but the request and response objects are passed along.

Code Example
GreetServlet.java 
@WebServlet("/greet")
public class GreetServlet extends HttpServlet {

    // Handles GET requests: reads query params, writes HTML response
    protected void doGet(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {

        // ---- REQUEST HANDLING: read input ----
        String name = req.getParameter("name");         // ?name=Riya
        String lang = req.getParameter("lang");         // ?lang=en

        // ---- RESPONSE HANDLING: write output ----
        res.setContentType("text/html");                // set MIME type
        PrintWriter out = res.getWriter();              // get writer

        out.println("<html><body>");
        out.println("<h2>Hello, " + name + "!</h2>");
        out.println("<p>Language: " + lang + "</p>");
        out.println("</body></html>");
    }
}

  • Mention both request methods: getParameter() and getWriter() — the former reads input, the latter writes output.

  • Mistake: calling getWriter() before setContentType() — the order matters and some containers will throw IllegalStateException.
💡 Mnemonic: "Read request params → Write response output."

Definition / Theory

Deploying a Servlet/JSP web application to Apache Tomcat involves creating a standard WAR (Web ARchive) file structure and placing it in the correct Tomcat directory. The process ensures the container can discover and run all servlets, JSPs, and static resources.

Step-by-step deployment process:

  • 1. Create project structure: Follow the standard Java EE directory layout: all Java source goes under src/main/java, JSP files and static resources under webapp (or WebContent).
  • 2. Write the deployment descriptor: Create WEB-INF/web.xml (or use @WebServlet annotations) to declare servlets, URL mappings, and configuration. If using annotations, web.xml is optional but must have version="6.0" metadata-complete="false".
  • 3. Compile and package: Use javac or a build tool (Maven/Gradle) to compile the .java files into .class files. Place them in WEB-INF/classes. Library JARs go in WEB-INF/lib. Package everything into a .war file using the jar command or Maven's package goal.
  • 4. Deploy to Tomcat: Copy the WAR file to $CATALINA_HOME/webapps/. Tomcat automatically detects the WAR, extracts it (into a directory with the same name), and deploys the application. Alternatively, deploy via the Tomcat Manager web app.
  • 5. Start/restart Tomcat: Run startup.sh (Linux/Mac) or startup.bat (Windows). Tomcat auto-deploys the WAR. Open http://localhost:8080/yourapp/ to access the application.

The key artifact is the WEB-INF directory — it is not accessible from the browser. It contains web.xml, compiled classes, and library JARs. Everything outside WEB-INF (JSP files, HTML, images, CSS) is directly accessible to the client.

    • WEB-INF/

      • classes/ (compiled .class files)
        • com/example/MyServlet.class
      • lib/ (JAR dependencies)
        • mysql-connector-java.jar

Definition / Theory

Servlet initialization parameters are configuration key-value pairs that are passed to a servlet at startup, before it handles any request. They allow you to externalize configuration from the Java code, making the servlet more flexible and reusable across different environments.

Two types of init parameters:

  • Servlet-level init parameters: Defined inside the <servlet> block in web.xml using <init-param>. They are accessible only to that specific servlet via getInitParameter(name) on the ServletConfig object. Common uses: database connection string, file paths, API keys, email addresses.
  • Context-level init parameters: Defined globally in <context-param> in web.xml. They are accessible to all servlets and JSPs in the application via getServletContext().getInitParameter(name). Common uses: application-wide settings like admin email, global upload directory.

Key methods:

  • getServletConfig().getInitParameter("paramName") — reads a servlet-level parameter.
  • getServletConfig().getInitParameterNames() — lists all servlet init parameter names.
  • getServletContext().getInitParameter("globalParam") — reads a context-level parameter.

Init parameters are resolved at deployment time. Changing a value in web.xml and reloading the application picks up the new value — no recompilation is needed. This separates configuration from code, a core principle of maintainable software engineering.

    urlPatterns = "/config",
    initParams = {
        @WebInitParam(name = "dbUrl",
                      value = "jdbc:mysql://localhost:3306/mydb")
    }
)
public class ConfigServlet extends HttpServlet {

    protected void doGet(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {

        // Read servlet-level init param
        String dbUrl = getServletConfig().getInitParameter("dbUrl");

        // Read context-level init param
        String adminEmail = getServletContext()
                              .getInitParameter("adminEmail");

        res.setContentType("text/html");
        PrintWriter out = res.getWriter();
        out.println("<h3>DB URL: " + dbUrl + "</h3>");
        out.println("<h3>Admin Email: " + adminEmail + "</h3>");
    }
}

Definition / Theory

Connecting a Java application to a database via JDBC follows six sequential steps. These steps are the same regardless of which database vendor you use (MySQL, PostgreSQL, Oracle) — only the JDBC URL and driver class change.

The six steps:

  • 1. Load the JDBC Driver: Using Class.forName("com.mysql.cj.jdbc.Driver"). This registers the driver with the DriverManager. In JDBC 4.0+ (Java 6+), this step is optional because drivers are auto-loaded from the classpath (META-INF/services/java.sql.Driver). However, explicitly loading is still common for compatibility.
  • 2. Create a Connection: Call DriverManager.getConnection(url, user, password). The URL format is vendor-specific: jdbc:mysql://host:port/dbname for MySQL, jdbc:postgresql://host:port/dbname for PostgreSQL. This returns a Connection object representing a socket-level connection to the database.
  • 3. Create a Statement: Use connection.createStatement() for static SQL, or connection.prepareStatement(sql) for parameterized queries (recommended to prevent SQL injection). A Statement object is the vehicle for sending SQL to the database.
  • 4. Execute the Query: Call statement.executeQuery(sql) for SELECT queries (returns ResultSet), or statement.executeUpdate(sql) for INSERT/UPDATE/DELETE (returns int row count).
  • 5. Process the ResultSet: Iterate through resultSet.next() and read column values with getString(), getInt(), getDate(), etc. Each next() call moves the cursor to the next row and returns false when no more rows remain.
  • 6. Close Resources: Always close ResultSet, Statement, and Connection in reverse order of creation. Use a try-with-resources block (Java 7+) to ensure automatic closure even if an exception occurs. Failing to close connections leads to resource leaks and database server connection exhaustion.

These six steps form the foundation of all JDBC-based database access in Java, whether in a standalone application or inside a Servlet/JSP web application.

Code Example

import java.sql.*;

public class JdbcSteps {

    public static void main(String[] args) {
        String url = "jdbc:mysql://localhost:3306/college";
        String user = "root";
        String pass = "password";

        // Step 1: Load driver (optional in JDBC 4+)
        try { Class.forName("com.mysql.cj.jdbc.Driver"); }
        catch (ClassNotFoundException e) { e.printStackTrace(); return; }

        // Steps 2–6 inside try-with-resources (auto-close)
        String sql = "SELECT id, name, marks FROM students WHERE marks > ?";

        try (Connection conn = DriverManager.getConnection(url, user, pass);  // Step 2
             PreparedStatement pstmt = conn.prepareStatement(sql);            // Step 3
        ) {
            pstmt.setInt(1, 80);                      // set parameter

            try (ResultSet rs = pstmt.executeQuery()) {                     // Step 4
                while (rs.next()) {                                         // Step 5
                    int id = rs.getInt("id");
                    String name = rs.getString("name");
                    int marks = rs.getInt("marks");
                    System.out.println(id + " | " + name + " | " + marks);
                }
            } // Step 6: rs closed automatically

        } catch (SQLException e) { e.printStackTrace(); }
        // Step 6: pstmt and conn closed automatically
    }
}

jQuery provides a set of effect methods that allow you to animate HTML elements with minimal code. The three fundamental methods — hide(), show(), and toggle() — control the visibility of elements on a web page.

Method details:

  • .hide(speed, easing, callback): Hides the matched elements by setting their display property to none (with an optional animation). Without parameters, it instantly hides. With speed ("slow", "fast", or milliseconds), it shrinks the element's width, height, and opacity simultaneously.
  • .show(speed, easing, callback): Reveals hidden elements by restoring their display property (usually block or inline). Like hide(), it supports animated reveal with optional speed and callback.
  • .toggle(speed, easing, callback): The smartest of the three — it checks the current visibility of the element. If visible, it hides it; if hidden, it shows it. This eliminates the need for if-else logic. With a boolean parameter .toggle(true) forces show and .toggle(false) forces hide.

Additional related effects: .fadeIn(), .fadeOut(), .fadeToggle() (opacity-only animation), and .slideDown(), .slideUp(), .slideToggle() (height-only animation). All these share the same optional speed, easing, and callback signature. The callback function is executed after the animation completes — useful for chaining further actions.

These methods are part of jQuery's Effects module and work in all modern browsers. They abstract away the complexities of CSS transitions and JavaScript timers, allowing developers to create polished UI interactions with a single line of code.

AJAX (Asynchronous JavaScript and XML) is a technique that allows web pages to communicate with the server asynchronously — sending requests and receiving responses without reloading the entire page. Despite "XML" in the name, AJAX commonly transfers JSON or plain text.

Key advantages of AJAX:

  • 1. No full page reloads: Only specific parts of the page are updated. The user can continue interacting with the rest of the page while the request processes in the background. This creates a fluid, desktop-like experience.
  • 2. Reduced bandwidth usage: Instead of re-sending the entire HTML page on every interaction, only small data payloads (JSON/XML) travel over the network. This saves bandwidth and reduces server load.
  • 3. Faster response time: Since only the relevant data is transferred (no re-rendering of the full page), the response feels instantaneous. The browser updates only the necessary DOM elements.
  • 4. Asynchronous processing: The browser does not freeze while waiting for the server response. Users can continue typing, scrolling, or clicking other elements while the AJAX call completes in the background.
  • 5. Improved interactivity: Features like autocomplete search, live form validation, real-time notifications, infinite scrolling, and chat are all powered by AJAX. These are now expected in modern web applications.
  • 6. Separation of concerns: AJAX promotes a clean API-based architecture where the frontend (HTML/JS) is decoupled from the backend (REST API). This allows teams to develop and scale independently.

Example scenario: A search box on an e-commerce site. Without AJAX, each keystroke would reload the page. With AJAX, the search suggestions appear dynamically below the input field as the user types, pulled from the server in real-time — this is Google Search's autocomplete experience.

In a Java web application, database connectivity is typically implemented through a combination of three components: a DBConnection utility class (to centralize connection management), JDBC API (for executing SQL), and Servlets (as the controller that ties user requests to database operations).

1. DBConnection Utility Class: A dedicated class that handles the boilerplate of loading the driver and creating Connection objects. It reads database credentials from init parameters (in web.xml) or a properties file. The class typically provides a static getConnection() method and a static closeConnection() method. This avoids duplicating the connection logic in every servlet.

2. JDBC in Servlets: The servlet's doGet() or doPost() method calls DBConnection.getConnection(), creates SQL statements, executes queries, processes ResultSet, and forwards the data to a JSP for display. Using PreparedStatement prevents SQL injection attacks.

3. DAO (Data Access Object) Pattern: A further improvement is to wrap all database operations in DAO classes (e.g. StudentDAO) that are called by the servlet. This separates data access logic from presentation/controller logic. The servlet only handles HTTP concerns (parameters, session, forwarding); the DAO handles SQL.

This three-layer design — Servlet (Controller) → DAO (Data Access) → DBConnection (Connection) — creates a clean, maintainable architecture where each component has a single responsibility. Changing the database vendor only requires updating the DBConnection class; the DAO and servlet code remain unchanged.

// ===== DBConnection.java (Utility) =====
public class DBConnection {
    private static final String URL =
        "jdbc:mysql://localhost:3306/mydb";
    private static final String USER = "root";
    private static final String PASS = "password";

    static {
        try { Class.forName("com.mysql.cj.jdbc.Driver"); }
        catch (ClassNotFoundException e) { e.printStackTrace(); }
    }

    public static Connection getConnection() throws SQLException {
        return DriverManager.getConnection(URL, USER, PASS);
    }

    public static void close(Connection conn, Statement stmt,
                              ResultSet rs) {
        try { if (rs != null) rs.close(); } catch (Exception e) {}
        try { if (stmt != null) stmt.close(); } catch (Exception e) {}
        try { if (conn != null) conn.close(); } catch (Exception e) {}
    }
}

// ===== UserDAO.java (Data Access Object) =====
public class UserDAO {
    public List<User> getAllUsers() throws SQLException {
        List<User> list = new ArrayList<>();
        String sql = "SELECT id, name, email FROM users";
        try (Connection conn = DBConnection.getConnection();
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery(sql)) {
            while (rs.next()) {
                list.add(new User(rs.getInt("id"),
                                  rs.getString("name"),
                                  rs.getString("email")));
            }
        }
        return list;
    }
}

// ===== UserServlet.java (Controller) =====
@WebServlet("/users")
public class UserServlet extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse res)
            throws ServletException, IOException {
        try {
            UserDAO dao = new UserDAO();
            List<User> users = dao.getAllUsers();
            req.setAttribute("userList", users);
            req.getRequestDispatcher("users.jsp").forward(req, res);
        } catch (SQLException e) {
            throw new ServletException(e);
        }
    }
}

An AJAX-based webpage for dynamic content loading is a page that fetches and displays new content from the server without a full page refresh. This technique is the foundation of modern single-page applications (SPAs) and powers features like infinite scroll, tabbed content, and load-more buttons.

Core components:

  • User trigger: A button click, scroll event, or timer initiates the content request.
  • JavaScript (AJAX call): Uses fetch() or XMLHttpRequest to send an asynchronous GET/POST request to a server endpoint. The request is non-blocking — the user can continue interacting with the page.
  • Server endpoint: A Servlet (or any server-side script) that processes the request and returns data — typically in JSON format (lightweight, easy to parse in JavaScript). The server does NOT return HTML — only raw data.
  • Client-side rendering: The JavaScript receives the JSON response, parses it, builds HTML elements dynamically, and inserts them into the DOM using innerHTML or DOM manipulation methods.
  • No page reload: The URL stays the same; only a specific section of the page (a <div> with a given ID) gets updated with fresh content.

This approach decouples the frontend from the backend. The frontend becomes a "consumer" of a REST API, and the backend does not care about HTML rendering — it only returns data. This is the same architecture used by React, Angular, and Vue.js applications.

Apache Tomcat is the most widely used open-source Servlet container and JSP engine. Its role in deploying Java web applications is fundamentally important — without Tomcat (or another container), a Java web application cannot run.

Role of Apache Tomcat:

  • Servlet Container: Tomcat implements the Jakarta Servlet specification. It manages the life cycle of servlets (loading, initialization, service, destruction) and handles the request-response pipeline. The container receives HTTP requests, creates request/response objects, calls the servlet's service() method, and sends the response back.
  • JSP Engine (Jasper): Tomcat includes Jasper, which translates .jsp files into Java servlet source code, compiles them, and executes them. This allows developers to write JSP pages without worrying about the translation step.
  • HTTP Web Server (Coyote): Tomcat's Coyote connector acts as a web server, listening for HTTP requests on a configurable port (default 8080). It can serve both static content (HTML, CSS, images) and dynamic content (through servlets/JSP).
  • Deployment Manager: Tomcat auto-deploys WAR files placed in the webapps/ directory. It also provides a Manager App (at /manager) for deploying, starting, stopping, and reloading applications without restarting the server.
  • Security & Realm: Tomcat supports declarative security via web.xml (login-config, security-constraint) and Realm implementations (JDBC, JNDI, Memory) for authentication and authorization.

Deployment process: Develop the application → Package as WAR → Copy WAR to $CATALINA_HOME/webapps/ → Start/restart Tomcat → Access at http://localhost:8080/appname/. Tomcat extracts the WAR, creates a context, and makes the application available.

A complete web application using Servlets (controller), Sessions (state management), and JDBC (database access) is a classic exam question that demonstrates all the core Java web technologies working together. The typical example is a Login/Profile application.

Application modules:

  • Login Page (login.jsp): A form with username and password fields. Submits POST to the LoginServlet.
  • LoginServlet (Controller): Reads the username/password from the request. Calls a UserDAO to validate credentials against the database using JDBC. If valid, creates an HttpSession, stores the user object in the session, and redirects to the profile page. If invalid, forwards back to the login page with an error message.
  • HttpSession (State): Once the user is logged in, the session object (req.getSession()) holds the user's data across multiple requests. This allows the profile page to display the logged-in user's information without re-querying the database on every request. The session persists until the user logs out or the session times out.
  • UserDAO (Data Access): Contains JDBC code to query the users table. Uses DBConnection.getConnection() and PreparedStatement to safely validate login credentials. Returns a User object (JavaBean) if found, or null if not.
  • Profile Page (profile.jsp): Reads the user from the session (session.getAttribute("user")) and displays their name, email, and other details. Includes a logout button.
  • LogoutServlet: Invalidates the session (session.invalidate()) and redirects to the login page.

This application architecture — JSP (View) → Servlet (Controller) → DAO (Model) → Database — is the textbook implementation of MVC (Model-2 Architecture) in Java web development.

            throws SQLException {
        String sql = "SELECT * FROM users WHERE email=? AND password=?";
        try (Connection conn = DBConnection.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {

            ps.setString(2, password);

Definition / Theory

Session tracking is the mechanism by which a web server maintains state across multiple HTTP requests from the same user. Since HTTP is stateless, each request is independent — without session tracking, the server would not know that two requests came from the same client. In Java Servlets, there are four standard session tracking techniques.

1. Cookies: The server sends a small piece of data (a session ID) in an HTTP response header: Set-Cookie: JSESSIONID=ABC123; Path=/; HttpOnly. The browser stores this cookie and sends it back with every subsequent request in the Cookie header. The servlet container uses the JSESSIONID to retrieve the correct HttpSession object via req.getSession(). Advantages: transparent to the developer, works automatically with HttpSession. Disadvantages: disabled if the user turns off cookies.

  • 2. URL Rewriting: The session ID is appended to every URL as a query parameter: http://host/app/page.jsp;jsessionid=ABC123 or ?jsessionid=ABC123. The server reads the ID from the URL. In Servlets, call response.encodeURL("page.jsp") to automatically append the session ID. Advantages: works even when cookies are disabled. Disadvantages: adds complexity to URLs, breaks if the user bookmarks or shares the URL.
  • 3. Hidden Form Fields: The session ID (or other state data) is stored in <input type="hidden" name="sessionId" value="ABC123"> within HTML forms. When the form is submitted, the hidden field value is sent to the server. Advantages: works without cookies. Disadvantages: only works with form submissions (not with anchor links), exposes session data in the HTML source.
  • 4. HttpSession API (the standard Java approach): The HttpSession object (obtained via req.getSession()) automatically uses either cookies (default) or URL rewriting (as fallback) to track the session. The developer uses session.setAttribute() and session.getAttribute() to store/retrieve any Java object. The container handles the low-level tracking transparently.

    • In modern Java web applications, HttpSession with cookies is the standard approach. The container automatically falls back to URL rewriting if cookies are not available. Session timeout is configured in web.xml (<session-config><session-timeout>30</session-timeout>) or programmatically via session.setMaxInactiveInterval().
